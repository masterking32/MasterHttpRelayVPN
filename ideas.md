## 1. داشبورد وب مدیریتی

یه UI محلی (localhost:PORT) که بشه بدون دست زدن به JSON همه چیز رو مدیریت کرد.

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان کامل. هیچ سرویس خارجی نیاز نیست. |
| ⏱️ **زمان** | ۴ / ۵ — نسخه اولیه ۱–۲ هفته، نسخه کامل با لاگ زنده و ویرایش config بیشتر. |
| 🎯 **تأثیر** | ۴ / ۵ — به شدت قابلیت استفاده رو برای کاربر غیر-فنی بالا می‌بره. |
| 🔗 **پیش‌نیاز** | ایده ۲ (آمار ترافیک) برای نمایش داده‌های زنده توصیه می‌شه اول پیاده بشه. |

**چی باید داشته باشه:**
- ویرایش config با فرم (نه JSON خام)
- نمایش ترافیک زنده (کدوم دامنه‌ها الان وصلن)
- toggle کردن allow/block/bypass برای هر دامنه با یه کلیک
- وضعیت relay (سبز/قرمز، latency)
- لاگ زنده با فیلتر

**پیاده‌سازی پیشنهادی:**
- یه HTTP server کوچک کنار proxy server (port جداگانه، default: 7891)
- Backend: aiohttp (چون already async هستیم)
- Frontend: HTML/JS ساده بدون framework (بدون build step)
- Auth: یه PSK ساده برای جلوگیری از دسترسی دیگران توی LAN

## 2. آمار و تحلیل ترافیک زنده

ردیابی اینکه چه حجمی از ترافیک از کجا رد شده و با چه latency.

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان کامل. SQLite داخل Python موجوده. |
| ⏱️ **زمان** | ۲ / ۵ — یه `StatsCollector` async در ۱–۲ روز قابل پیاده‌سازیه. |
| 🎯 **تأثیر** | ۴ / ۵ — به کاربر کمک می‌کنه تنظیمات رو بهینه کنه و مشکلات رو سریع پیدا کنه. |
| 🔗 **پیش‌نیاز** | ندارد. مستقل پیاده می‌شه. |

**متریک‌هایی که ارزش داره:**
- بایت‌های ارسال/دریافت per domain
- تعداد request per domain
- latency میانگین relay vs direct
- نرخ خطا per domain
- پیک ساعتی

**چرا مهمه:** میشه دید کدوم relay بهتره، کدوم دامنه‌ها کند هستن، و آیا exit node واقعاً کمک می‌کنه.

**پیاده‌سازی پیشنهادی:**
- یه `StatsCollector` async در `proxy_server.py` که در حافظه نگه می‌داره
- Export به فرمت JSON از طریق داشبورد
- Optional: ذخیره سازی به SQLite برای تاریخچه

## 3. روتینگ هوشمند خودکار

پروکسی خودش تشخیص بده که کدوم دامنه‌ها بلاک هستن و route بهینه رو انتخاب کنه — بدون اینکه کاربر دست بزنه.

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان کامل. |
| ⏱️ **زمان** | ۳ / ۵ — نسخه پایه (fallback + cache) حدود ۳–۵ روز؛ learning mode پیچیده‌تره. |
| 🎯 **تأثیر** | ۵ / ۵ — صفر کانفیگ برای کاربر. مهم‌ترین بهبود تجربه کاربری پروژه. |
| 🔗 **پیش‌نیاز** | ندارد. ولی اگه ایده ۴ (Relay Pool) هم باشه خیلی بهتر کار می‌کنه. |

**ایده:**
- وقتی connection برای دامنه‌ای fail می‌شه، اتوماتیک relay رو امتحان کن
- نتیجه رو cache کن (این دامنه → این route)
- هر N ساعت cache رو expire کن و دوباره چک کن
- یه "learning mode" که در background دامنه‌های fail رو تست می‌کنه

**چرا این خفنه:** کاربر نیازی نداره هیچ config بزنه — پروکسی خودش می‌فهمه.

## 4. Load Balancing بین چند relay

الان یه Apps Script URL داریم. اگه چند تا داشتیم و بین‌شون load balance می‌کردیم:

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان. Google Apps Script رایگانه و می‌شه با چند Google account چند تا deploy کرد. |
| ⏱️ **زمان** | ۳ / ۵ — `RelayPool` class با round-robin در ۳–۵ روز؛ weighted + health check بیشتر. |
| 🎯 **تأثیر** | ۴ / ۵ — پایداری و uptime رو به شدت بالا می‌بره. Rate limit دیگه مشکل نیست. |
| 🔗 **پیش‌نیاز** | چند Google account (رایگان) + هر کدوم باید Apps Script جداگانه deploy شده باشه. |

**مزایا:**
- اگه یکی rate limit خورد، بقیه کار می‌کنن
- latency کمتر (موازی probe + سریع‌ترین انتخاب می‌شه)
- اگه یکی down شد، failover خودکار

**پیاده‌سازی پیشنهادی:**
- `relay_urls` به جای `relay_url` در config (آرایه)
- یه `RelayPool` class که:
  - health check هر relay رو periodic انجام بده
  - بر اساس latency + error rate وزن‌دهی کنه
  - round-robin ساده برای شروع

## 5. DNS رمزنگاری‌شده (DoH/DoT)

الان DNS lookup‌ها plaintext هستن — ISP می‌بینه داری کجا DNS می‌زنی.

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان. سرویس‌های DoH مثل Google DNS و Cloudflare رایگان هستن. |
| ⏱️ **زمان** | ۲ / ۵ — یه `DoHResolver` wrapper در ۱–۲ روز پیاده می‌شه. |
| 🎯 **تأثیر** | ۴ / ۵ — یه حفره امنیتی مهم رو می‌بنده. خیلی از ISPها روی DNS فیلتر می‌کنن. |
| 🔗 **پیش‌نیاز** | دسترسی به یکی از این سرویس‌های رایگان: `dns.google`، `cloudflare-dns.com`، یا `dns.quad9.net`. |

**راه‌حل:**
- یه DNS resolver داخلی که از DNS-over-HTTPS (DoH) استفاده کنه
- Provider‌های پیشنهادی: `dns.google`, `cloudflare-dns.com`, `dns.quad9.net`
- Cache کردن DNS جواب‌ها برای سرعت

**پیاده‌سازی پیشنهادی:**
- کتابخانه `aiodns` + یه `DoHResolver` wrapper
- قابل تنظیم در config: `"doh_provider": "https://dns.google/dns-query"`
- Fallback به system DNS اگه DoH fail شد

## 6. Kill Switch و WebRTC Leak Protection

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان. Windows Firewall و iptables داخل سیستم‌عامل هستن. |
| ⏱️ **زمان** | ۴ / ۵ — هر پلتفرم API جداگانه داره. حداقل ۱–۲ هفته برای پوشش ۳ سیستم. |
| 🎯 **تأثیر** | ۴ / ۵ — برای کاربران حساس (روزنامه‌نگار، فعال مدنی) یه ویژگی حیاتیه. |
| 🔗 **پیش‌نیاز** | دسترسی admin/root روی سیستم کاربر (برای تغییر firewall rules). |

**Kill Switch:**
اگه پروکسی down بشه یا crash کنه، traffic نباید مستقیم از شبکه رد بشه. Kill switch همه ترافیک رو قطع می‌کنه تا پروکسی دوباره بیاد بالا.

- روی Windows: Windows Firewall rules موقت
- روی Linux: iptables rules
- یه watchdog process که وضعیت رو monitor کنه

**WebRTC Leak Protection:**
مرورگرها از طریق WebRTC می‌تونن IP واقعی رو leak کنن حتی با پروکسی.
- یه PAC file خودکار generate کنه که WebRTC رو هم cover کنه
- داکیومنت: "این extension رو نصب کن"

## 7. پروفایل‌های چندگانه

یه نفر ممکنه در محیط‌های مختلف باشه: خونه، محل کار، سفر.

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان کامل. |
| ⏱️ **زمان** | ۱ / ۵ — `--profile` flag در `main.py` در کمتر از نیم روز پیاده می‌شه. |
| 🎯 **تأثیر** | ۳ / ۵ — QoL خوبه ولی بیشتر برای power userهاست. |
| 🔗 **پیش‌نیاز** | ندارد. مستقل پیاده می‌شه. |

**ایده:**
- چند config profile: `config.home.json`, `config.work.json`, `config.travel.json`
- دستور سریع switch: `python main.py --profile work`
- یا auto-detect بر اساس IP network (اگه subnet این بود → این profile)

**مثال کاربرد:**
- Profile "کار": فقط دامنه‌های مشخص allow (whitelist mode)
- Profile "خونه": همه چیز relay بشه
- Profile "سفر": exit node فعال، kill switch روشن

## 9. پشتیبانی از پروتکل‌های بیشتر

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان. WinDivert برای transparent proxy open-source و رایگانه. |
| ⏱️ **زمان** | ۵ / ۵ — هر کدوم به تنهایی ۱ ماه+ زمان می‌بره. پیچیده‌ترین آیتم لیست. |
| 🎯 **تأثیر** | ۵ / ۵ — اگه پیاده بشه، از یه HTTPS proxy به یه full VPN تبدیل می‌شه. |
| 🔗 **پیش‌نیاز** | برای UDP tunnel: باید Apps Script یا exit node از UDP/WebSocket پشتیبانی کنه (تغییر بزرگ در relay). برای Transparent Proxy: نصب WinDivert روی Windows یا دسترسی root روی Linux. |

**UDP / QUIC Tunneling:**
الان فقط TCP داریم. خیلی از برنامه‌ها (بازی‌ها، VOIP، QUIC-based HTTP/3) UDP نیاز دارن.
- یه UDP-over-WebSocket یا UDP-over-HTTPS tunnel
- سخت‌تره ولی خیلی پرکاربرد

**Transparent Proxy:**
بدون اینکه کاربر تنظیمات پروکسی رو عوض کنه، تمام ترافیک سیستم رد بشه.
- روی Linux: iptables redirect
- روی Windows: WinDivert
- این essentially یه full VPN می‌شه

## 12. Relay Pool و تشخیص relay مرده

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان. Apps Script رایگانه و چند Google account هم رایگانه. |
| ⏱️ **زمان** | ۲ / ۵ — health check پایه در ۱–۲ روز؛ auto-provision با clasp کمی بیشتر. |
| 🎯 **تأثیر** | ۴ / ۵ — پایداری رو زیاد می‌کنه. وقتی relay rate limit می‌خوره دیگه قطع نمی‌شیم. |
| 🔗 **پیش‌نیاز** | چند Google account (رایگان). برای auto-provision: نصب `clasp` (ابزار CLI گوگل برای Apps Script). |

**Health Check خودکار:**
- هر N دقیقه یه probe request به relay بزن
- اگه جواب نداد یا کند بود → اخطار بده
- اگه چند relay داریم → به relay بعدی switch کن

**Auto-provision:**
- اسکریپت که خودکار چند تا Apps Script deploy کنه (از طریق `clasp`)
- هر relay توی یه Google account مجزا
- Rotation خودکار بین‌شون

## 14. ادغام با Tor

برای کاربرانی که maximum anonymity می‌خوان:

| | |
|---|---|
| 💰 **هزینه مالی** | ۱ / ۵ — رایگان کامل. Tor Project رایگانه. |
| ⏱️ **زمان** | ۱ / ۵ — در کمتر از یه روز پیاده می‌شه. |
| 🎯 **تأثیر** | ۳ / ۵ — برای ۹۵٪ کاربران لازم نیست ولی برای موارد حساس بسیار مهمه. سرعت کم می‌شه. |
| 🔗 **پیش‌نیاز** | نصب Tor روی سیستم کاربر (رایگان، از torproject.org). بدونش SOCKS5 روی پورت 9050 وجود نداره. |

**Chain:** Client → این پروکسی → Tor → Destination

- یه `tor_mode` در config که relay رو روی Tor SOCKS5 (127.0.0.1:9050) هدایت کنه
- Tor باید جداگانه نصب باشه
- این یعنی ISP نه IP اصلی می‌بینه نه destination

**نکته:** سرعت خیلی کم می‌شه ولی برای موارد حساس مفیده.

**مشکل الان:** کاربر باید دستی از GitHub بکشه و update کنه.

**راه‌حل:**
- یه دستور `python main.py --update` که:
  1. آخرین release رو از GitHub API چک کنه
  2. اگه نسخه جدید بود، دانلود و اعمال کنه
  3. Apps Script code رو هم در صورت نیاز redeploy کنه
- یه پیام در startup اگه نسخه جدید موجود باشه

## یادداشت‌های متفرقه

- **IPv6:** بررسی کنیم آیا relay روی IPv6 هم کار می‌کنه
- **Rate limiting:** اضافه کردن per-client rate limit برای وقتی proxy در LAN share شده
- **Prometheus metrics endpoint:** برای کسایی که Grafana دارن
- **Config validation:** schema validation برای config.json با پیام خطای واضح
- **Hot reload:** reload کردن config بدون restart با SIGHUP
